# .main.lifecycle.md — team1 platform lifecycle reference

Canonical lifecycle for the team1 platform. Every unit maps its work onto these phases. Each agent's skill file declares which phases it **leads (L)**, **supports (S)**, or **reviews (R)**.

## Master lifecycle (7 phases)

```
1. Planning & Requirements
2. Architecture & Design
3. Implementation & Build
4. Testing & QA
5. Deployment & Release
6. Monitoring & Incident Response
7. Analysis & Feedback
        │
        └──► feeds back into Planning (next cycle)
```

### Phase definitions

| # | Phase | Purpose | Exit criteria (artifact **AND** MBO — co-equal) |
|---|-------|---------|---------------|
| 1 | Planning & Requirements | Validate feasibility, define scope, set MBO-aligned targets. | Scope doc approved **AND** metric targets loaded from `assignments/teamelite-2025.json` with baselines recorded. |
| 2 | Architecture & Design | Produce design docs, ADRs, diagrams, data models. | ARB sign-off, NFRs shown achievable **AND** architecture-review-coverage MBO on target (100% of major changes). |
| 3 | Implementation & Build | Build features/services per design. | Code merged via PR, unit/integration tests green **AND** technical-debt-ratio MBO on target (<15%) **or** a planned-gap declaration accepted per the aligned rule below. |
| 4 | Testing & QA | Functional, performance, security, accessibility validation. | QA gate passed **AND** bug-escape-rate MBO on target (<5%) **or** accepted remediation per the aligned rule. |
| 5 | Deployment & Release | Ship to production via CI/CD. | Deployment success, release notes published **AND** deployment-success-rate MBO on target (>99%). |
| 6 | Monitoring & Incident Response | Observe SLOs; respond to incidents. | SLOs observed **AND** the owning unit's MBO metrics within target (loaded from `assignments/<project>.json` — e.g. SaaS: 99.9% uptime / sub-200ms; Cloud: 95% cost util; Sec: zero critical vulns); off-target triggers a `RemediationIntent` back to the relevant phase, not a silent pass. |
| 7 | Analysis & Feedback | Postmortems, metric review, learnings. | Retrospective complete, improvements backlogged **AND** MBO gaps from phases 3–6 are mandatory inputs to the next cycle's Phase 1. |

> *Specific MBO targets per unit are defined in `assignments/<project>.json` and the per-unit section below. The examples in the table above (99.9% uptime, <15% debt, <5% bug escape, >99% deploy) are drawn from the SaaS Development Unit; other units substitute their own metrics.*

### Aligned rule (MBO miss does not block forever)

An MBO miss is resolved by exactly one of three paths. **Path (c) is a per-phase default-available declaration, not an escalation** — the declaring agent records the planned gap in its progress report as a normal field; the unit Manager reviews accumulated declared gaps **at Phase 7**, not gate-by-gate:

- **(a) Remediate in-phase** — the agent closes the gap before the phase exits.
- **(b) Backward intent** — `lifecycle-management` emits a `RemediationIntent` to the relevant earlier phase (e.g. Monitoring → Implementation).
- **(c) Planned-gap declaration** — the agent declares the gap as a per-phase field in its progress report (routine, cheap, no escalation required). Declared gaps are collected and reviewed by the Manager at Phase 7 Analysis; accepted gaps become mandatory Phase-1 inputs next cycle.

**Why (c) is routine, not exceptional:** co-equal MBO gates, applied gate-by-gate with heavyweight acceptance, punish ambition — agents play safe to avoid debt-ratio spikes. Making (c) a default-available declaration keeps the gate honest (the gap is *declared*, never hidden) while letting agents take real swings and report debt openly. The Manager's review happens once, at Phase 7, over the full set of declared gaps — not as a per-gate bottleneck.

**Enforcement:** the `lifecycle-management` service blocks no phase transition until both gates are resolved — artifact gate passed **and** MBO either on-target, remediated (a/b), or declared (c). A declared gap resolves the MBO gate; it does not skip it.

## Cross-unit handoff rules

- **Handoffs require an artifact** — no phase exits on a verbal handoff. The producing agent logs the artifact in `00_workspace/working_files/ARTIFACT_INDEX.md` and references its ID in their progress report.
- **Cross-unit dependencies** (e.g. SaaS ↔ Cloud, ML/Ops ↔ Data Science) are coordinated via the relevant Architect/Manager, not agent-to-agent directly.
- **Incidents** during Monitoring escalate to the owning unit's Manager; if multi-unit, escalate to both Managers (see escalation triggers in each skill file).
- **Feedback loop** — Phase 7 outputs (postmortems, metric gaps) are mandatory inputs to the next cycle's Phase 1 Planning.

## Progress reporting cadence

- **Daily** during Implementation, Testing, Deployment phases.
- **Weekly** during Planning, Architecture, Monitoring, Analysis phases.
- **Per-incident** for any production-impacting event, regardless of phase.
- Format: copy `00_workspace/working_files/progress/_template.md`.
