## Risk Zone 1: Redlock Lease Timeout (§11.1) — CRITICAL

### The Problem
v1.0 runs everything inside the 45s lock: checkout → apply → **tests** → commit → push.
If test suite exceeds 45s, lock auto-expires, another agent acquires it, both agents operate on same branch → **silent corruption**.

### v1.1 Solution: Two-Tier Architecture
```
Tier 1 (Locked, <5s budget):
  git checkout + apply batch + git add + git commit + git push → RELEASE LOCK

Tier 2 (Async, after release):
  Create git worktree clone → run full test suite
  
  If pass: emit EditApplied
  If fail: emit TestFailed{commitSha}
    → edit-coordinator acquires lock briefly
    → git revert <sha>
    → git push
    → release lock
    → emit EditReverted
```

### Why This Fixes It
- Lock is held only for git operations that take **seconds**, not minutes
- Tests run async after release, no timeout risk
- Failed tests trigger retroactive revert (agent receives revert in next seed context)
- **Eliminates silent corruption entirely**

### Impact on Codebase
**M3 Code at §6.2 is now WRONG.** The `runWorktreeValidation()` function is called INSIDE the locked block:

```typescript
// CURRENT (v1.0) — WRONG AFTER v1.1
const lock = await redlock.acquire([resource], leaseTtl);
try {
  await git.checkout(branch);
  for (const intent of intents) { /* apply edits */ }
  
  // ❌ THIS MUST MOVE OUT (it's the test suite)
  const testResult = await runWorktreeValidation(branch, paths);
  if (!testResult.passed) { throw; }
  
  await git.commit(commitMsg);
  await git.push('origin', branch);
} finally {
  await lock.release();
}
```

**Required M3 Refactor:**
```typescript
// NEW (v1.1) — CORRECT
const lock = await redlock.acquire([resource], leaseTtl);
try {
  await git.checkout(branch);
  for (const intent of intents) { /* apply edits */ }
  await git.commit(commitMsg);
  await git.push('origin', branch);
  const commitSha = (await git.revparse(['HEAD'])).trim();
  
  // ✅ Emit event; Tier 2 runner will consume
  await bus.publish('intents:TestNeeded', {
    commitSha, branch, paths, featureSlug, traceId
  });
} finally {
  await lock.release();
}

// NEW ASYNC HANDLER (Tier 2)
async function handleTestNeeded(intent) {
  const testResult = await runWorktreeValidation(intent.branch, intent.paths);
  if (!testResult.passed) {
    await bus.publish('intents:TestFailed', { commitSha: intent.commitSha, ... });
  } else {
    await bus.publish('intents:EditApplied', { commitSha: intent.commitSha, ... });
  }
}

// NEW REVERT HANDLER (called by TestFailed consumer)
async function handleTestFailed(intent) {
  const lock = await redlock.acquire([resource], leaseTtl);
  try {
    await git.checkout(intent.branch);
    await git.revert([intent.commitSha]);
    await git.push('origin', intent.branch);
  } finally {
    await lock.release();
  }
  await bus.publish('intents:EditReverted', { commitSha: intent.commitSha, ... });
}
```

### Verification Caveat (Post-M2)
- Profile Tier 1 latency (git ops only) on target monorepo
- Measure P95; if >5s, investigate worktree-cached checkouts
- Document final profile in `orchestrator/infra/redis-keyspaces.md`
- Until then, 5s budget is an estimate, not validated

## Risk Zone 2: Backtrack Phase Logic (§11.2) — DEADLOCK RISK

### The Problem
v1.0 phase map: `7: 6` (Phase 7 → Phase 6 on failure)
Risks:
1. Phase 7 (Analysis) is terminal — can't loop back to work phases
2. Undetected infinite loops at non-terminal phases (e.g., 4 → 3 → 2 → 1 → ... → 4)

### v1.1 Solution: Two Fixes

**Fix A — Phase 7 does not backtrack**
```typescript
function resolveBackwardPhase(currentPhase: number, backtrackCount: number): number {
  if (currentPhase === 7) {
    throw new Error('PHASE_7_NO_BACKTRACK');
    // Feature held in "blocked" state
    // Manager must extend deadline OR accept gaps
  }
  // ... rest of map
}
```

Rationale: Phase 7 = Analysis (retrospective, not code). If it fails, the problem is process (incomplete retrospective), not implementation. Re-running Implementation won't fix a missing retrospective. Manager intervention required.

**Fix B — Backtrack counter escalation**
```typescript
if (backtrackCount >= 3) {
  throw new Error('BACKTRACK_LIMIT_EXCEEDED');
  // Emit PhaseEscalation instead of backward intent
  // Manager decides: accept planned-gap or kill feature
}
```

Prevents infinite loops at non-terminal phases. After 3 backtracks on same phase, no more automatic regressions — escalate to Manager.

### Updated Phase Map
```typescript
const map: Record<number, number> = {
  1: 1, // Planning (can't go back)
  2: 1, // Architecture → Planning
  3: 2, // Implementation → Architecture
  4: 3, // Testing → Implementation
  5: 3, // Deployment → Implementation
  6: 3, // Monitoring → Implementation
  // 7: REMOVED — throws above
};
```

### Impact on Codebase
**M4 Code at §7.1 is now WRONG.** Current map has `7: 6`. Must be removed and replaced with:

```typescript
export async function evaluateGateCheck(
  featureSlug: string,
  currentPhase: number,
  artifactCompleted: boolean,
  backtrackCount: number  // ← NEW PARAM
) {
  const mboStatus = await queryMboMetrics(featureSlug, currentPhase);
  const bothPassed = artifactCompleted && mboStatus.ok;
  const gapDeclared = mboStatus.gapDeclared;

  if (bothPassed || gapDeclared) {
    // Advance
    await updatePhaseDatabase(featureSlug, currentPhase + 1, 0); // reset backtrack counter
    await bus.publish('intents:phase-gate-check', { ..., status: 'advanced' });
  } else {
    // Backtrack with counter
    const newBacktrackCount = backtrackCount + 1;
    
    if (currentPhase === 7) {
      // Phase 7 special: don't backtrack, block feature
      await updatePhaseDatabase(featureSlug, currentPhase, newBacktrackCount);
      await bus.publish('intents:phase-gate-check', {
        ..., status: 'blocked', reason: 'Phase 7 analysis incomplete'
      });
    } else if (newBacktrackCount >= 3) {
      // Too many backtracks; escalate
      await bus.publish('intents:phase-escalation', {
        featureSlug, phase: currentPhase, backtrackCount: newBacktrackCount
      });
    } else {
      // Normal backtrack
      const backtrackPhase = resolveBackwardPhase(currentPhase);
      await updatePhaseDatabase(featureSlug, backtrackPhase, newBacktrackCount);
      await bus.publish('intents:phase-gate-check', {
        ..., status: 'backtrack', targetPhase: backtrackPhase
      });
    }
  }
}
```

### Verification
- Backtrack counter must be persisted in lifecycle-management SQLite
- Phase backtrack mapping must be unit-tested (all edges, all dead-ends validated)
- Phase 7 blocking must be visible in orchestrator-api (feature shows `verdict: 'blocked'`)

## Risk Zone 3: Multi-Feature Merge Coordination (§11.3) — SCOPE CUT

### The Problem
v1.0: Auto-detect feature dependencies via path-overlap heuristic.
Risk: Path overlap ≠ semantic conflict. Auto-detection is fragile, M5 effort heavily underestimated.

### v1.1 Solution: Explicit API Only
```
POST /features/:slug/dependsOn/:otherSlug
  Authorization: Manager token required
  Body: { depends_on: "other-slug" }
  
  Response: 201 Created
```

Stored in lifecycle-management SQLite:
```sql
CREATE TABLE feature_dependencies (
  feature_slug TEXT NOT NULL,
  depends_on_slug TEXT NOT NULL,
  declared_at INTEGER NOT NULL,
  PRIMARY KEY (feature_slug, depends_on_slug)
);
```

Merge rule: Feature B cannot enter Phase 5 (Deployment) until every feature in B.dependsOn has Phase 7 = passed.

### Why This Fixes It
- **Explicit > Implicit:** Managers declare real dependencies, not guessed heuristics
- **CRUD problem, not research:** Storage + query logic, trivial scope
- **M5 is now buildable:** Clear APIs, no blocker on path-analysis research
- **Future: Auto-detection deferred to M7+** as optional suggestion layer (never enforced)

### Impact on Codebase
**M5 scope reduced by ~60%.** Remove auto-detection heuristics; add:
1. `/features/:slug/dependsOn/:otherSlug` endpoint in orchestrator-api (Manager-authority check)
2. `feature_dependencies` table in lifecycle-management SQLite
3. Query in MergeCoordinator: `SELECT depends_on_slug FROM feature_dependencies WHERE feature_slug = ?`
4. Merge gate: Block Phase 5 entry if any `depends_on` feature is not Phase 7

## Open Questions After v1.1

1. **Tier 2 Service Location:** Does the async test runner live in edit-coordinator or separate service?
   - **Option A:** Inside edit-coordinator (simpler, tight coupling)
   - **Option B:** Separate service consumed by edit-coordinator (cleaner, adds complexity)
   - **Recommendation:** Option A for M5. Post-launch, if test parallelism needed, extract to Option B.

2. **TestFailed → Revert Latency:** How quickly should failed tests trigger revert? Immediate, or aggregate batches?
   - **Recommendation:** Immediate per test failure (no batching). Delays confuse agents about causality.

3. **Backtrack Escalation Workflow:** When Phase Escalation is emitted, what does the Manager do?
   - **Options:** (a) Accept planned-gap via token, (b) extend deadline, (c) kill feature
   - **Recommend:** Document workflow in Phase 7 runbook before M4 code starts

4. **Tier 2 Test Scope:** Can we safely run only `npm test -- --findRelatedTests <paths>` or must we run full suite per commit?
   - **Recommendation:** Start with full suite (conservative). Post-launch, measure if full suite is bottleneck; then optimize to affected-path-only.
